Let's dive deep into the Strategy Pattern with a simple, complete code example. I'll use Java, but the concepts apply to any object-oriented language.

## What is the Strategy Pattern?

The Strategy Pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable. It lets the algorithm vary independently from the clients that use it. The pattern **replaces conditional logic (if-else or switch) with polymorphism**, adhering to the Open/Closed Principle: classes are open for extension but closed for modification.

### Key Participants

- **Strategy** – an interface (or abstract class) common to all supported algorithms.
- **ConcreteStrategy** – implementations of the Strategy interface, each providing a specific algorithm.
- **Context** – maintains a reference to a Strategy object and delegates the algorithm execution to it. The context may also define an interface that lets the strategy access its data.

## Simple Example: Payment Methods

Imagine an e‑commerce application where users can pay with different methods: Credit Card, PayPal, or Bitcoin. Instead of writing a `PaymentProcessor` class with a giant `if-else` chain, we use the Strategy Pattern.

### Step 1: Define the Strategy Interface

```java
// The common interface for all payment strategies
interface PaymentStrategy {
    void pay(double amount);
}
```

### Step 2: Implement Concrete Strategies

```java
class CreditCardStrategy implements PaymentStrategy {
    private String cardNumber;
    private String expiryDate;
    private String cvv;

    public CreditCardStrategy(String cardNumber, String expiryDate, String cvv) {
        this.cardNumber = cardNumber;
        this.expiryDate = expiryDate;
        this.cvv = cvv;
    }

    @Override
    public void pay(double amount) {
        // Here you'd integrate with a real payment gateway
        System.out.println("Paid $" + amount + " using Credit Card ending with " 
                           + cardNumber.substring(cardNumber.length() - 4));
    }
}

class PayPalStrategy implements PaymentStrategy {
    private String email;
    private String password; // In real life, you'd use tokens, not raw passwords

    public PayPalStrategy(String email, String password) {
        this.email = email;
        this.password = password;
    }

    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " using PayPal account " + email);
    }
}

class BitcoinStrategy implements PaymentStrategy {
    private String walletAddress;

    public BitcoinStrategy(String walletAddress) {
        this.walletAddress = walletAddress;
    }

    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " using Bitcoin wallet " + walletAddress);
    }
}
```

### Step 3: Create the Context

The context uses a `PaymentStrategy` to perform the payment. It doesn’t know which concrete strategy it’s using – it just calls `pay()`.

```java
class ShoppingCart {
    private List<Item> items;
    private PaymentStrategy paymentStrategy;

    public ShoppingCart() {
        this.items = new ArrayList<>();
    }

    public void addItem(Item item) {
        items.add(item);
    }

    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.paymentStrategy = strategy;
    }

    public void checkout() {
        double total = items.stream().mapToDouble(Item::getPrice).sum();
        paymentStrategy.pay(total);
        items.clear(); // clear cart after payment
    }
}
```

For completeness, here’s a simple `Item` class:

```java
class Item {
    private String name;
    private double price;

    public Item(String name, double price) {
        this.name = name;
        this.price = price;
    }

    public double getPrice() { return price; }
}
```

### Step 4: Use the Pattern

Now we can choose the payment method at runtime without any `if-else` inside the shopping cart.

```java
public class Main {
    public static void main(String[] args) {
        ShoppingCart cart = new ShoppingCart();
        cart.addItem(new Item("Laptop", 1200.00));
        cart.addItem(new Item("Mouse", 25.50));

        // Pay with Credit Card
        cart.setPaymentStrategy(new CreditCardStrategy("1234-5678-9012-3456", "12/25", "123"));
        cart.checkout();

        // Pay with PayPal
        cart.setPaymentStrategy(new PayPalStrategy("user@example.com", "secret"));
        cart.checkout();
    }
}
```

**Output:**

```
Paid $1225.5 using Credit Card ending with 3456
Paid $1225.5 using PayPal account user@example.com
```

## How It Replaces If-Else

Without the Strategy Pattern, the `checkout()` method might have looked like:

```java
public void checkout(String paymentType, Map<String, String> details) {
    double total = ...;
    if ("credit".equals(paymentType)) {
        // credit card logic
    } else if ("paypal".equals(paymentType)) {
        // paypal logic
    } else if ("bitcoin".equals(paymentType)) {
        // bitcoin logic
    }
}
```

This approach violates the Open/Closed Principle: adding a new payment method requires modifying the existing `checkout()` method. With the Strategy Pattern, we simply add a new `ConcreteStrategy` and pass it to the context – no existing code changes.

## Benefits in Detail

1. **Eliminates conditional statements** – makes the code cleaner and easier to read.
2. **Follows Open/Closed Principle** – new strategies can be added without touching existing classes.
3. **Encapsulates algorithm families** – each strategy is isolated, making it easier to test and maintain.
4. **Runtime flexibility** – strategies can be swapped dynamically based on user input, configuration, or state.
5. **Single Responsibility** – the context focuses on its core job (e.g., managing a cart), while each strategy handles a specific payment logic.

## When to Use the Strategy Pattern

- When you have multiple related classes that differ only in their behavior (algorithms).
- When you need different variants of an algorithm (e.g., sorting algorithms, compression algorithms, validation rules).
- When you want to avoid exposing complex algorithm data structures to the client.
- When a class defines many behaviors and these appear as multiple conditional statements in its operations.

## Real-World Analogy

Think of a GPS navigation app. It can offer different routes (fastest, shortest, scenic, avoid highways). Each route is a strategy. The app (context) doesn’t need to know the details of how each route is calculated; it simply calls the `calculateRoute()` method on the current strategy.

## Conclusion

The Strategy Pattern is a powerful tool to replace conditional logic with polymorphism. It makes your code more flexible, easier to extend, and aligns with fundamental design principles. The pattern is widely used in frameworks (e.g., Java’s `Comparator` is a classic strategy) and is a must‑know for any developer aiming to write clean, maintainable code.
